# OpenTicker 产品分析与需求拆解

## 1. 问题定义

OpenCode 本身擅长交互式运行、单次 `run` 和服务化 `serve`，但没有面向“计划性触发”的心跳调度层。用户在这些场景里会遇到明显缺口：

- 需要预制一批任务，让 OpenCode 在固定时间自动执行。
- 需要一次性任务，在某个明确时间点触发。
- 需要延时任务，在若干分钟或若干小时后补跑。
- 需要后台稳定执行，而不是依赖用户一直开着终端窗口。
- 需要一个足够易用的界面，否则最后还是回到手改 cron 和脚本。

OpenTicker 的定位就是补这一层：它不是替代 OpenCode，而是把 OpenCode 变成“可编排、可计划、可后台运行”的执行引擎。

## 2. 产品目标

### 核心目标

- 用 TUI 提供一个一屏可管理的任务控制台。
- 让用户能在 1 分钟内创建首个任务。
- 同时覆盖循环任务、一次性任务、延时任务三种典型触发方式。
- 提供后台 daemon 和系统 service，尽量解决“关掉终端就不跑”的问题。
- 保持安装和使用足够轻量，优先采用 npm CLI 交付。

### 非目标

- 不直接实现一个新的 LLM agent。
- 不复刻 OpenCode 全部参数面板。
- 不做重型 Web 控制台。
- 不做复杂 DAG 编排、分布式 worker、审计平台。

## 3. 目标用户

- 有固定巡检、日报、代码审查、回归检查任务的开发者。
- 希望通过 OpenCode 定时处理仓库工作的个人用户。
- 不愿意手写 `cron`、`launchd`、`systemd` 配置的终端用户。
- 喜欢键盘优先、极客风格工具的人。

## 4. 使用场景

### 场景 A：循环心跳任务

用户想让 OpenCode 每小时扫描一次仓库，输出简短变化总结，同时通过 `--attach` 尽量复用预热服务，减少冷启动成本。

### 场景 B：一次性任务

用户有一个明确的发布时间点，希望 OpenCode 在当天某个时间自动执行发布前检查。

### 场景 C：延时任务

用户刚发起一轮长耗时流程，希望 45 分钟后自动再跑一次 follow-up 检查。

### 场景 D：预制任务包

用户想先准备一批默认任务，之后导入到另一台机器或另一个项目。

## 5. 功能需求

### P0

- 支持 `cron` 循环任务。
- 支持 `once` 一次性任务。
- 支持 `delay` 延时任务。
- 支持 TUI 查看、创建、编辑、启停、删除、手动运行任务。
- 支持本地 JSON 配置持久化。
- 支持后台 daemon 轮询调度。
- 支持任务日志落盘。
- 支持 `doctor` 检查依赖和运行状态。

### P1

- 支持 macOS `launchd` 和 Linux `systemd --user` 安装 service。
- 支持 GitHub 仓库直接安装的 npm CLI 包结构。
- 支持样例任务导入导出。
- 支持预设模板，降低首次配置成本。

### P2

- 更丰富的任务参数模板。
- 环境变量模板化。
- webhook/通知集成。
- 任务执行统计视图。

## 6. 交互要求

- 首屏就能看到任务列表、当前状态、下一次执行时间。
- 所有核心操作都要有快捷键。
- 创建任务必须走“尽量少字段”的 wizard。
- 默认提供示例 preset，避免空白界面。
- 界面风格要偏黑客终端，不走商务后台风。

## 7. 体验要求

- 开箱即用：安装后直接执行 `openticker` 即可进入 TUI。
- 轻依赖：避免原生模块和复杂构建链。
- 好看但不过度装饰：深色背景、霓虹线框、键盘优先。
- 易安装：支持 `npm install -g` 或从 GitHub 直接安装。

## 8. 技术约束

- 调度层必须通过 `opencode run` 完成任务执行。
- 后台执行必须不依赖 TUI 常驻。
- 配置文件需要人类可读。
- 日志必须落盘，便于排查。
- 依赖尽量控制在纯 JS 范围内，避免安装原生编译链。

## 9. 当前版本交付清单

- 可运行的 Node.js npm CLI 项目。
- TUI 控制台。
- daemon 管理命令。
- service 安装器。
- README、产品说明、架构说明、贡献文档。
- 示例任务配置文件。
